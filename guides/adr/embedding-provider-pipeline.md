# ADR 019: Embedding Provider Pipeline (Internal ONNX + External Providers)

## Status
Proposed

## Context
TrueRAG currently expects vectors to be supplied for vector/hybrid retrieval lanes, while document chunking remains out of scope.  
To reduce integration burden and support enterprise deployment models, TrueRAG needs a configurable embedding pipeline for:

- document chunk embeddings during ingestion
- query embeddings during retrieval/search

The system must preserve modular-monolith boundaries and avoid coupling the core to a single vendor.

## Decision
TrueRAG will introduce a dedicated Embeddings module with provider abstraction and two provider families:

1. Internal ONNX provider family (self-hosted local inference)
2. External API provider family (managed providers)

Initial internal default model:
- `BAAI/bge-small-en-v1.5` (384 dimensions, 512 max tokens)

Additional internal selectable models:
- `BAAI/bge-base-en-v1.5` (768 dimensions, 512 max tokens)
- `sentence-transformers/multi-qa-MiniLM-L6-cos-v1` (384 dimensions, 512 max tokens)

External provider interface targets:
- OpenAI
- Voyage AI
- Cohere
- DeepInfra

Configuration and invariants:
- Each `tenant_id + app_id` scope uses exactly one active embedding provider/model pair at a time.
- Query and document embeddings must be generated by the same configured model family/space for that scope.
- Mixing vectors from different models in one app scope is rejected.
- Changing provider/model requires explicit re-embedding workflow before activation.
- Chunking remains out of scope; only embedding generation is in scope.

## Consequences

Positive:
- Supports fully self-hosted and managed-provider deployment options.
- Keeps retrieval quality consistent by enforcing model-space consistency.
- Preserves current external-vector mode while adding optional internal generation.

Trade-offs:
- Adds operational complexity (model lifecycle, warmup, provider health, cost controls).
- Requires compatibility validation and migration workflow when changing models.

## Non-Goals
- Chunking/parsing
- Vector search/hybrid fusion redesign
- Reranking
- LLM response generation/conversation memory changes

## Rollout Strategy
1. Internal ONNX provider with `bge-small-en-v1.5` as default.
2. Additional internal ONNX model options.
3. External provider adapters (OpenAI, Voyage, Cohere, DeepInfra).
4. App-scope embedding contract enforcement and re-embedding governance.
