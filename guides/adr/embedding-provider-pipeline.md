# ADR 019: Embedding Provider Pipeline (Internal ONNX + External Providers)

## Status
Proposed

## Context
TrueRAG needs one embedding pipeline that supports self-hosted and managed-provider deployments while preserving modular-monolith boundaries and scoped isolation.

The platform must support:

- document chunk embeddings during ingestion
- query embeddings during retrieval/search

Document chunking remains upstream/out of scope.

## Decision
TrueRAG uses a dedicated `TrueRag.Embeddings` module with provider abstraction and two provider families:

1. Internal ONNX provider family (self-hosted local inference)
2. External API provider family (managed providers)

Initial internal default model:
- `BAAI/bge-small-en-v1.5` (384 dimensions, 512 max tokens)

Additional internal selectable models:
- `BAAI/bge-base-en-v1.5` (768 dimensions, 512 max tokens)
- `sentence-transformers/multi-qa-MiniLM-L6-cos-v1` (384 dimensions, 512 max tokens)

External provider rollout:
- OpenAI is the first supported external provider.
- Provider abstraction stays reusable for future providers (Voyage AI, Cohere, DeepInfra), but those remain disabled until explicitly implemented and validated.

Scope and invariants:
- Each `tenant_id + app_id + collection_id` scope uses exactly one active embedding provider/model pair at a time.
- Query and document embeddings must be generated by the same configured model family/space for that scope.
- Mixing vectors from different models in one app scope is rejected.
- Changing provider/model requires explicit re-embedding workflow before activation.
- Chunking remains out of scope; only embedding generation is in scope.

Credential and ownership rules:
- External provider credentials (for example OpenAI API key) are supplied by upstream platform configuration, bound in `TrueRag.Host` configuration surfaces, and injected through module options.
- Credentials are never accepted from ingestion/retrieval payloads.
- Host/environment/secret-store configuration is the only source of truth for provider credentials.
- Provider logs/metrics must redact secrets and raw text inputs.

Execution contract alignment (hard gate):
- Sync ingestion (`/api/v1/ingest/sync`) is client-precomputed-vector-only and never performs embedding execution.
- Async ingestion (`/api/v1/ingest/async`) is pipeline-orchestrated embedding execution (internal ONNX or configured external provider) and does not accept client-provided vectors.
- Retrieval uses the same scoped descriptor as ingestion for vector-space compatibility.

## Consequences

Positive:
- Supports fully self-hosted and managed-provider deployment options.
- Keeps retrieval quality consistent by enforcing model-space consistency.
- Preserves strict sync client-vector mode while enabling async provider orchestration.
- Provides a clear secret-management boundary (host-owned credentials only).

Trade-offs:
- Adds operational complexity (model lifecycle, warmup, provider health, cost controls).
- Requires compatibility validation and migration workflow when changing models.
- Requires explicit upstream credential provisioning and rotation workflows.

## Non-Goals
- Chunking/parsing
- Vector search/hybrid fusion redesign
- Reranking
- LLM response generation/conversation memory changes

## Rollout Strategy
1. Internal ONNX provider with `bge-small-en-v1.5` as default.
2. Additional internal ONNX model options.
3. External provider adapter for OpenAI (first external provider in production path).
4. Async worker orchestration path uses provider registry + scoped descriptor resolution.
5. Future provider onboarding (Voyage/Cohere/DeepInfra) through same abstraction after dedicated validation.
6. App-scope embedding contract enforcement and re-embedding governance.

## Ingestion Execution Mode Alignment
Per ADR 021:
- Sync ingestion remains client-precomputed-vector mode (no embedding execution in sync path).
- Async ingestion is pipeline-orchestrated embedding mode (WAL/queue/worker execution) and does not accept client vectors.
