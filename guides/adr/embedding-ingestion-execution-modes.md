# ADR 021: Embedding Ingestion Execution Modes (Sync Client-Precomputed, Async Pipeline-Orchestrated)

## Status
Proposed

## Context
TrueRAG has two ingestion paths:
- `POST /api/v1/ingest/async`: WAL + queue + worker execution
- `POST /api/v1/ingest/sync`: direct write path

With Phase 8 embedding support, we must keep deterministic contracts between ingestion and retrieval.

## Decision
TrueRAG keeps both ingestion paths with strict, non-overlapping contracts:

1. Sync path (`/ingest/sync`)
- remains available and backward-compatible
- accepts chunk text + **precomputed vectors** from caller
- does **not** run any internal/external embedding generation
- represents client-managed embedding space

2. Async path (`/ingest/async`)
- remains the durability-first path (WAL append before background processing)
- accepts chunks/text from caller
- embedding vectors are generated by TrueRAG pipeline only (worker orchestration)
- provider execution is selected by scope configuration (internal ONNX or external provider adapter)
- caller-provided vectors are not part of async contract

3. Query-time embedding
- for sync/client-managed embedding spaces, caller supplies query vector (+ query text for hybrid/text lanes)
- for async/pipeline-managed embedding spaces, TrueRAG generates query vector from query text
- query and document vectors must use the same active scoped descriptor

## Consequences

Positive:
- preserves current sync expectations and latency profile
- keeps embedding latency off API hot path for async ingestion
- aligns with WAL/NATS operational model for large-volume ingestion
- removes ambiguous mixed-mode behavior in async

Trade-offs:
- behavior differs by endpoint and retrieval contract must be documented clearly
- validation must strictly reject contract violations for each mode

## Guardrails
- Sync request with missing vectors must fail validation when sync is configured as precomputed-only.
- Async request with caller-provided vectors must fail validation.
- Async worker embedding failures must be observable and recoverable via queue/WAL replay patterns.
- Batch embedding + batch upsert should be used in async worker flow for throughput.
